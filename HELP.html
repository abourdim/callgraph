✦ Getting Started البداية

  1. Start the launcher: run ./start.sh from the project directory. The interactive menu checks all dependencies and shows their status before launch.
  2. Open the app: navigate to http://localhost:7411 (or the port you chose in the launcher).
  3. Index your project: type the absolute path to your C source directory in the index box — or click Browse to navigate the filesystem — then click ◈ Index Project.
  4. Watch the stream: progress messages appear live as the 7-pass analyzer processes your files. A 200-file project takes about 2–3 seconds.
  5. Explore: the full call graph appears. Click any function in the sidebar list or any node in the graph to enter trace mode. Use the Analysis panel for dead code, change impact, heatmap, path finding, and intersection traces.
Session persistence: After the first index, results are stored in localStorage for 24 hours. Reloading the page restores the graph instantly — no re-index needed until your source changes.
Re-index vs New project: Click ⟳ Re-index to reparse the same path after code changes. Click ＋ New to clear everything and index a different project.

✦ Frequently Asked Questions الأسئلة الشائعة

What kinds of C project does this work with?

Any C project: bare-metal embedded firmware, RTOS applications, HAL/driver libraries, communication stacks, bootloaders. The parser is tree-sitter which handles preprocessor-heavy embedded C well. Only .c files are parsed — .cpp files are skipped.

How long does indexing take?

About 0.3–0.5 s per 10 source files on typical hardware. Measured on real projects: EVIC (69 files, 552 functions) = 0.3 s, HMI SDK (206 files, 570 functions) = 2.4 s. After the first index, localStorage caching makes reloads instant for 24 hours.

Some functions are missing from the graph.

tree-sitter follows explicit call_expression AST nodes. Functions invoked only through function pointers stored in structs or arrays will not appear as edges — the pointer assignment is visible but the call target is not statically resolvable. External library functions (called but not defined in your .c files) appear as gray "external" nodes — hide them with the Ext button to reduce visual noise.

My graph has 500+ nodes and is slow or unreadable.

Use the Depth slider (sidebar) to limit to 2–3 hops from root functions. Or use Trace mode on a specific function — it only renders that function's immediate callers and callees and stays fast at any project size. The Module view (⬡) is always fast and gives the architectural overview without node overload.

Is my source code sent anywhere?

No. Everything runs entirely on your machine. The Flask server binds to localhost only. No analytics, no telemetry, no network requests of any kind are made. Your code never leaves your computer.

Can I use this on a remote build server?

Yes. SSH port forwarding works cleanly:

ssh -L 7411:localhost:7411 user@build-server
# Then open http://localhost:7411 locally

Or set PORT=8080 before launching to use a different port.

The race detector found zero races. Is that correct?

Possibly yes. The detector requires: (1) a function with an ISR-convention name that writes a file-scope global, and (2) a separate non-ISR function that reads or writes that same global, and (3) no critical section entry call detected in the non-ISR function. If your ISRs use non-standard naming, or your shared data goes through RTOS queues/mailboxes rather than raw globals, the detector correctly finds nothing.

The RTOS panel is empty.

The RTOS panel detects FreeRTOS and CMSIS-RTOS v2 API calls by name. If your project wraps these APIs behind macros (e.g. #define OS_TASK_CREATE xTaskCreate), tree-sitter sees only the macro name and cannot match it. The panel covers 44 direct API call names — see the RTOS section below for the full list.

PNG export is black or blank.

PNG export renders the SVG through an HTML Canvas element, which some browsers restrict via cross-origin policies, producing a black image. Use SVG export instead — it is always reliable, resolution-independent, and smaller. To convert to PNG: open in Inkscape and export, or run rsvg-convert -f png -o out.png graph.svg.

The session is not restored after page reload.

localStorage must be enabled in your browser. Private / incognito mode disables it. Check your browser privacy settings. If localStorage is unavailable, re-index from the same path — the analyzer is fast enough that this is not a burden.

✦ Tracing a Function تتبع دالة

  1. Type the function name in the Trace function box — autocomplete activates after 1 character. Use ↑/↓ to navigate, Enter to confirm.
  2. Or click any node in the full graph — it enters trace mode immediately.
  3. The graph re-renders: callers on the left, the traced function highlighted in the centre, callees on the right.
  4. Click any node in the trace graph to trace that function — history updates automatically.
  5. Navigate history with ◀ / ▶ or Alt+← / Alt+→. The breadcrumb trail below the graph bar shows the full session path — click any crumb to jump back.
  6. Press Escape or click ✕ Trace to return to the full graph.
The info panel (click the ⓘ icon on any node) shows full details: file, line, all callers, all callees. Click any name in the info panel to trace it.

✦ Dead Code Detector الكود الميت

  1. In the Analysis section of the sidebar, click ☠ Dead code.
  2. The results panel lists all functions with zero project-level callers, sorted by file.
  3. Click any function name in the list to trace it in the graph.
  4. Click Show in graph — dead functions are highlighted red, all others dim to 20% opacity.
  5. Click Export list — downloads a .txt file with function name, file path, and line number for each dead function.
Auto-excluded entry points (never flagged as dead): main, ISR_*, irq_*, *_irq, *_isr, interrupt_*, handler_*, init_*, _init, startup, reset_handler, _start, setup.
Limitation: Functions called only via function pointer appear dead even when live at runtime. In callback-heavy code (state machines, dispatch tables), cross-check results manually.

✦ Change Impact Analysis تحليل أثر التغيير

  1. In Analysis, click ⚡ Impact.
  2. Paste the names of the functions you changed — one per line or comma-separated. Autocomplete is not active here; spelling must match exactly.
  3. Click ⚡ Analyse impact.
  4. Results show total affected caller count and a per-module breakdown.
  5. Click Show in graph — red nodes = your changed functions, orange nodes = transitively affected callers.
  6. Click ◁ Edit to return to the input and adjust the function list.
Algorithm: Reverse BFS — starting at each changed function and walking upward through all callers recursively. The blast radius is every function that can transitively reach any of your changes through any call chain.

✦ Complexity Heatmap خريطة التعقيد

  1. In Analysis, click 🌡 Heatmap.
  2. The top 10 highest-scoring functions are listed with score, file, and out/in/depth components.
  3. Click any function name to trace it.
  4. Click Apply to graph — every node is recoloured on a green → yellow → red gradient based on its score relative to the project maximum.
  5. Click Clear to restore the original node colours.
Scoring formula
score = (out_degree × 2) + in_degree + call_depth

Out-degree is weighted 2× because fan-out determines how many downstream callers you must audit and test. In-degree measures how many callers depend on this function. Call-depth is the BFS distance from the nearest root function (one with no project callers). Colour thresholds: green < 33% < yellow < 66% < red of the project maximum score.

✦ Call Path Finder مُوجِد المسار

  1. In Analysis, click ⇢ Path find.
  2. Enter a From function (source) and a To function (target). Both validate against known function names.
  3. Click ⇢ Find paths.
  4. Results appear ranked shortest-first — each shows hop count and the full function sequence.
  5. Click any path row to render it as a step-numbered graph.
  6. Click any node in the path graph to trace it further.
No path found? The call graph is directed — A calling B does not mean B calls A. Try swapping From and To. Maximum search depth: 12 hops. Maximum results returned: 20 paths.

✦ Intersection Trace تقاطع المسارات

  1. In the Intersection trace sidebar section, enter Function A and Function B.
  2. Click ⊕ Intersect.
  3. The graph renders only the functions that appear in both call trees — the shared callees of A and B.
  4. Click Clear to return to the full graph.
Algorithm: Full descendant BFS from both functions independently, then set intersection of all reachable nodes. Useful for: "what code is shared between the CAN stack and the USB stack?" or "what does both the init path and the error recovery path touch?"

✦ Global Variable Dependencies تبعيات المتغيرات العامة

  1. Click Globals in the graph bar (green) — appears after indexing a project that has file-scope variables.
  2. All detected globals are listed. Volatile variables appear first. Summary shows total count and volatile count.
  3. Each row shows: write count (red W badge), read count (green R badge), read-modify-write count (purple RW badge).
  4. Click any variable name — the graph highlights writers red, readers green, RW functions purple, all others dim.
  5. Expand a variable row to see the per-function list. Click any function name to trace it.
  6. Click ✕ in the panel header or press Escape to close and restore the graph.
Write detection: direct assignment (x = val), compound assignment (x |= mask, x &= ~bit, x += n), increment/decrement (x++, --x). RW: read-modify-write (the left side of a compound assignment is both read and written). Read: any right-hand-side or argument use of the global identifier.

✦ Race Condition Analysis تحليل حالات السباق

  1. Click Races in the graph bar (red) — appears only when races are detected.
  2. Candidates are ranked: HIGH first, then MEDIUM, then LOW. A summary shows totals by severity.
  3. Click any candidate row — the ISR function highlights red, the non-ISR function highlights orange in the graph.
  4. Close the panel to restore the graph.
SeverityCondition
HIGHVolatile global written by an ISR-named function, accessed (read or written) by a non-ISR function with no critical section entry call detected
MEDIUMNon-volatile global — same unprotected access pattern
LOWGlobal accessed across ISR/non-ISR boundary, but the non-ISR function body contains a detected critical section entry call

✦ RTOS Architecture بنية نظام التشغيل الفوري

  1. Click RTOS in the graph bar (purple) — appears when RTOS API calls are detected.
  2. Each task creator is listed with its observed relationships: creates tasks, sends to queues, receives from queues, takes mutexes/semaphores, gives mutexes/semaphores, delays.
  3. Click any task function name to trace it in the call graph.
  4. The Shared Objects section lists all detected queues, mutexes, and semaphores with every function that uses each one.

✦ Peripheral Register Map خريطة سجلات الأجهزة

  3. Function names are coloured: red = write, green = read, purple = read-modify-write.
  4. Click any peripheral name — all functions touching it are highlighted in the graph.
  5. Click any function name in the panel to trace it.

✦ Analysis Reference مرجع التحليل

☠ Dead Code Detector

Algorithm: Finds project-type nodes where the effective in-degree from other project nodes is 0 (no project function calls them). External function callers do not count.

Excluded: main, ISR_*, irq_*, *_irq, *_isr, interrupt_*, handler_*, init_*, _init, startup, reset_handler, _start, setup

Known limitation: Functions called only via function pointer appear dead. Cross-check in callback-heavy code.

⚡ Change Impact Analysis

Algorithm: Reverse BFS from each changed function upward through the caller graph. Terminates when no more callers are reachable or when a node has already been visited.

Output: Total affected count, grouped by module, with graph overlay (red = changed, orange = affected callers).

🌡 Complexity Heatmap
score = (out_degree × 2) + in_degree + call_depth

call_depth: BFS distance from the nearest root (no project callers) to this function.

Colour mapping: green = score < 33% of project max · yellow = 33–66% · red = > 66%

⇢ Call Path Finder

Algorithm: DFS with a visited-set to avoid cycles. Backtracks at depth 12. Stops when 20 paths are found. Results sorted by path length ascending.

Rendered graph: clicking a path shows each function numbered in sequence order.

⊕ Intersection Trace

Algorithm: Full descendant BFS from Function A and from Function B independently, producing two sets. The intersection (nodes in both sets) is rendered as a subgraph.

Use cases: "What code is shared between the USB driver and the CAN driver?" · "What do both the boot path and the OTA path call?"

SymptomResolution
"No functions found"Verify the path contains .c files. Run Diagnostics in the launcher menu to test the parser directly.
